# QNAP NAS 照片/視頻歸檔系統

## 概述

本文檔描述了在 QNAP NAS 上運行的照片和視頻歸檔系統。該系統的核心目標是將從手機備份的照片和視頻（位於 `upload` 目錄）按照統一的規範整理到 `archive` 目錄，實現：

- **可追溯**：通過標準化命名快速定位文件
- **不丟失**：智能處理重複文件，確保內容不丟失
- **可維護**：統一的目錄結構，便於長期管理

---

## 系統環境

### 基本配置

- **設備**: QNAP NAS
- **SSH 連接**: 使用你自己的 NAS 主機、用戶和認證方式
- **本地私有信息**: 請存放在未納入版本控制的私有文件中，不要寫入公開文檔

### QNAP 權限系統的特殊性

QNAP 的權限管理與標準 Linux 系統有顯著差異，這是所有操作必須注意的核心問題：

#### 1. Admin 賬號的限制
- `admin` 是系統保留賬號，**無法直接 SSH 登錄**
- 必須通過普通用戶 + sudo 方式獲取 admin 權限

#### 2. ACL 權限自動還原
- QNAP 的 ACL 系統會**自動還原**手動修改的文件權限
- 直接 `chmod` 或 `chown` 的修改會被覆蓋
- **解決方案**：所有涉及權限的操作都必須以 admin 身份執行

#### 3. Sudo 使用方式
- 標準 sudo 需要密碼輸入
- 腳本中需要管道密碼或使用 `-S` 參數
- 執行多個命令時使用 `-i` 切換到 admin 環境

### 可用工具

QNAP 基於嵌入式 Linux，系統工具有限，但可通過 Entware 擴展。

#### 系統內置工具

| 工具 | 狀態 | 用途 |
|------|------|------|
| bash | ✅ | 腳本執行 |
| find | ✅ `/usr/bin/find` | 文件查找 |
| stat | ✅ `/usr/bin/stat` | 文件屬性獲取 |

#### Entware 工具（位於 `/opt/bin/`）

| 工具 | 狀態 | 用途 |

- 無 python，複雜邏輯需用 bash 或 perl 實現
- ffmpeg 版本較舊（3.3.6），但基本元數據提取功能正常

## 核心設計理念

### 為什麼需要歸檔系統？

手機備份的照片和視頻通常存在以下問題：

1. **命名混亂**：IMG_001, VID_20230101, 微信視頻_123 等
2. **來源多樣**：iCloud、Android、微信、WhatsApp 等
3. **時間信息不一致**：文件名時間與實際拍攝時間不符
4. **大量重複**：多次備份、編輯副本、壓縮版本混雜

### 設計決策

#### 目錄結構：YYYY/MM 而非 YYYY-MM-DD
- **原因**：平衡可瀏覽性和目錄數量
- 月級別分類足夠日常查找，避免日級別目錄過多
- 特殊需求可通過搜索解決

#### 文件名包含大小信息
- **原因**：時間相同的情況下，大小是天然的去重因子
- 相同時間 + 相同大小 = 極大概率是同一文件
- 不同大小 = 肯定是不同文件（編輯版本、壓縮版本）

#### 重複文件的處理策略
- **內容相同**（MD5 一致）：添加序號後綴 `_1`, `_2`
- **內容不同**：添加 hash 後綴，保留所有版本
- **設計哲學**：寧可保留冗餘，不可丟失任何內容

## 目錄結構

### Upload 目錄（輸入）

臨時存放從各來源備份的原始文件：

```
/share/ssd/upload/
├── icloud-photos/          # iCloud 照片導出
├── icloud-photos-1/        # 拆分後的子目錄
├── icloud-photos-2/
├── iphonexs-YYYY-MM/       # iPhone 按月拆分
└── ...                     # 其他來源
```

**特點**：
- 結構混亂，多來源混雜
- 命名不統一
- 可能包含大量重複
- 文件時間信息不可靠

### Archive 目錄（輸出）

標準化歸檔的永久存儲位置：

```
/share/archive/
├── photos/                 # 照片歸檔
│   └── YYYY/
│       └── MM/
│           └── YYYYMMDD_HHMMSS_大小.ext
├── videos/                 # 視頻歸檔
│   └── YYYY/
│       └── MM/
│           └── YYYYMMDD_HHMMSS_大小.ext
├── pngs/                   # PNG 圖片歸檔
│   └── YYYY/
│       └── MM/
│           └── YYYYMMDD_HHMMSS_大小.png
└── thumbnails/             # 縮略圖歸檔
    └── YYYY/
        └── MM/
            └── YYYYMMDD_HHMMSS_大小.jpg
```

**特點**：
- 統一的 YYYY/MM 結構
- 標準化的文件命名
- 重複文件有明確的處理規則
- 可追溯、可維護

## 文件命名規範

### 標準格式

```
YYYYMMDD_HHMMSS_文件大小.擴展名
```

### 字段說明

| 字段 | 格式 | 說明 | 示例 |
|------|------|------|------|
| YYYYMMDD | 8位數字 | 文件日期（年月日） | 20230115 |
| HHMMSS | 6位數字 | 文件時間（時分秒） | 143052 |
| 文件大小 | 數字 | 文件大小（字節） | 12345678 |
| 擴展名 | 小寫 | 文件擴展名 | mp4, mov, jpg |

### 命名示例

```
20230115_143052_12345678.mp4      # 標準文件
20230115_143052_12345678_1.mp4    # 第一個重複（內容相同）
20230115_143052_12345678_2.mp4    # 第二個重複（內容相同）
20230115_143052_12345678_a1b2c3d4.mp4  # 內容不同，hash 後綴
```

### 重複文件處理規則

當目標文件名已存在時：

1. **計算 MD5 hash**（取前 8 位）
2. **比較 hash**：
   - **相同**：內容一樣，添加序號 `_1`, `_2`, `_3`...
   - **不同**：內容不同，保留兩個版本，新文件添加 hash 後綴

## 時間信息提取策略

時間是文件命名的核心，但不同來源的可靠性不同。

### 時間來源優先級

1. **EXIF/元數據**（最可靠）
   - 照片：EXIF 原始拍攝時間
   - 視頻：ffmpeg 提取的 creation_time

2. **文件系統時間**（次可靠）
   - mtime（修改時間）
   - 注意：複製文件時可能改變

3. **文件名解析**（最不可靠）
   - 從文件名提取日期（如 VID_20230115_...）
   - 僅作為最後手段

### 視頻元數據提取（ffmpeg）

視頻文件的最佳時間來源是嵌入的元數據：

- **命令**：`ffmpeg -i file 2>&1 | grep creation_time`
- **格式**：`2024-11-02T11:49:25.000000Z`
- **成功率**：約 88% 的視頻有元數據
- **無元數據情況**：多為微信視頻（v- 前綴文件名）

### 照片 EXIF 提取（exiftool）

- **工具**：exiftool（需確認安裝）
- **字段**：DateTimeOriginal 或 CreateDate
- **注意**：編輯過的照片可能丟失 EXIF

### 無時間信息處理

當無法獲取任何可靠時間時：
- 使用 1970-01-01 作為基準
- 文件大小作為去重依據
- 需要人工審核（在 1970 目錄中）

## 腳本最佳實踐

### 權限處理

**核心原則**：所有文件操作都應以 admin 身份執行

#### 正確方式

1. **上傳腳本**：使用普通用戶上傳到 /tmp 或用戶目錄
2. **執行腳本**：通過 sudo 以 admin 身份執行

#### 執行模式

- **單命令**：`echo '密碼' | sudo -S command`
- **多命令**：`echo '密碼' | sudo -S -i -u admin bash -c 'commands'`
- **腳本文件**：`echo '密碼' | sudo -S bash /path/to/script.sh`

### 腳本上傳和執行流程

1. **本地編寫/修改腳本**
2. **上傳到 NAS**：使用 scp 傳輸到 /tmp
3. **SSH 登錄**
4. **以 admin 執行**
5. **檢查結果**
6. **清理臨時文件**

### 腳本設計原則

#### 冪等性
- 腳本可以重複執行
- 已處理的文件不會重複處理
- 錯誤不會導致數據損壞

#### 錯誤處理
- 記錄失敗的文件
- 繼續處理後續文件
- 提供可審計的日誌

#### 性能考慮
- 批量操作（10萬+ 文件需要數小時）
- 避免不必要的子進程
- 只在需要時計算 hash

### 處理大量文件的技巧

- **分批處理**：按目錄或時間範圍分割任務
- **進度顯示**：定期輸出進度信息
- **中斷恢復**：設計支持從中斷點繼續
- **日誌記錄**：記錄處理過程便於審計

## 特定類型文件處理經驗

### 視頻文件（mp4, mov, 3gp）

**挑戰**：
- 文件名時間與拍攝時間不一致
- 不同設備的元數據格式不同

**解決方案**：
- 優先使用 ffmpeg 提取 creation_time
- 無元數據時 fallback 到 mtime
- 微信視頻（v- 前綴）通常無元數據

### PNG 文件

**挑戰**：
- PNG 沒有標準 EXIF
- 多為截圖或編輯後的圖片

**解決方案**：
- 直接使用 mtime
- 注意：編輯時間 ≠ 原始時間

### 縮略圖（thumbnails）

**挑戰**：
- 數量巨大（10萬+）
- 原始結構複雜（/100/2023/02/...）

**解決方案**：
- 批量重組目錄結構
- 統一命名格式
- 注意磁盤 I/O 性能

## 常見問題和排查

### 權限被拒絕
- **原因**：未以 admin 身份執行
- **解決**：使用 sudo 並確保密碼正確

### 文件時間是 1970 年
- **原因**：無法獲取有效時間信息
- **解決**：檢查是否有元數據，必要時手動修正

### 處理中斷
- **原因**：網絡斷開、SSH 超時
- **解決**：使用 nohup 或 screen 長時間運行

### 重複文件過多
- **原因**：多次備份同一內容
- **解決**：歸檔後可用腳本批量去重（風險較高）

## 維護和擴展

### 添加新的文件類型

1. 確定時間提取策略
2. 確定目標目錄（photos/videos/pngs/...）
3. 編寫/修改處理腳本
4. 以 admin 身份測試執行
5. 驗證結果

### 定期維護

建議定期執行：
- 檢查 archive 目錄完整性
- 清理 upload 目錄已處理文件
- 驗證命名規範符合率
- 檢查異常時間的文件（1970 年）

### 監控指標

- 未處理文件數量（upload 目錄）
- 不符合命名規範的文件數
- 重複文件比例
- 存儲空間使用情況
